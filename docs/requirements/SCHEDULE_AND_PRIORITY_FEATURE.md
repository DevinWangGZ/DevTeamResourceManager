## 背景与目标

### 痛点

1. **优先级缺失**：当前任务系统无优先级概念，PM 只能口头沟通紧急程度，排期不透明。
2. **排期不可视**：开发人员只知道自己"有几个任务在手"，不知道应该先做哪个、何时能做完。
3. **PM 优先级滥用风险**：一旦引入优先级，若没有成本约束，PM 会倾向于把所有任务都标为高优先级，失去意义。
4. **项目进展不透明**：项目经理无法一眼看清项目所有任务的时间分布与交付节奏。

### 目标

- 通过**经济成本**约束 PM 合理使用高优先级
- 让开发人员**一眼看清**今天/本周该做什么
- 让项目经理**一眼看清**项目各任务的时间线分布

### FR-01 任务优先级字段

#### 描述

在任务创建/编辑表单中新增**优先级**字段，在任务列表、详情页中展示。

#### 优先级定义

| 优先级 | 含义               | 默认值 | 颜色标识 |
|--------|--------------------|--------|----------|
| P2     | 常规，正常流程处理 | ✅ 是  | 灰色     |
| P1     | 较紧急，优先排入   | 否     | 橙色     |
| P0     | 最紧急，插队处理   | 否     | 红色     |

#### 业务规则

- 任务创建时默认为 **P2**
- 优先级仅在任务处于**草稿、已发布**状态时可修改（认领后不可更改，避免中途加塞）
- 优先级在任务集市列表、个人工作台任务列表、项目任务列表中均显示

#### 验收标准

- [ ] 任务创建表单中有优先级选择器，默认选中 P2
- [ ] 优先级在任务列表和详情页用色标清晰展示
- [ ] 已认领及之后状态的任务，优先级字段只读
- [ ] 任务集市支持按优先级筛选

### FR-02 优先级费率溢价

#### 描述

引入优先级费率溢价机制，当 PM 选择高于默认（P2）的优先级时，该任务的**产值计算单价**按比例上浮，从而形成经济约束，抑制 PM 无节制拉高优先级。

#### 溢价规则

| 优先级 | 溢价系数 | 说明                             |
|--------|----------|----------------------------------|
| P2     | ×1.00    | 无溢价，标准单价                 |
| P1     | ×1.10    | 上浮 10%，适度提升成本           |
| P0     | ×1.20    | 上浮 20%，显著提升成本，慎重使用 |

#### 产值计算公式（更新）

```
任务产值 = 实际投入人天 × 开发人员序列单价 × 优先级溢价系数
```

> 原公式：`任务产值 = 实际投入人天 × 开发人员序列单价`
> 本次更新后溢价系数为乘数项。

#### 用户感知设计

- 任务创建表单中，当 PM 切换优先级时，**实时展示"溢价说明"提示**：
  - 选择 P1 时：`⚠️ 选择P1优先级将使该任务产值上浮 10%`
  - 选择 P0 时：`⚠️ 选择P0优先级将使该任务产值上浮 20%，请谨慎使用`
- 在任务详情页和产值明细中，清晰展示溢价系数和溢价金额

#### 验收标准

- [ ] 产值计算公式已更新，包含溢价系数
- [ ] 任务创建/编辑时切换优先级有实时溢价提示
- [ ] 任务详情页展示溢价系数和溢价金额
- [ ] 产值明细报表中新增"优先级溢价系数"列
- [ ] 项目总产值计算正确反映各任务的溢价

### FR-03 认领时自动排期（融合优先级与并发）

#### 描述

在原有排期逻辑（§2.6 任务排期）基础上，融合优先级规则、配合人排期同步、并发任务支持，形成新的**综合排期算法**。

---

#### 3.1 核心概念

##### 串行任务 vs 并发任务

| 模式       | 含义                                                     | 日历显示         |
|------------|----------------------------------------------------------|------------------|
| **串行**   | 任务在队列中顺序执行，前一任务结束后才开始下一任务       | 任务块不重叠     |
| **并发**   | 多个任务时间区间重叠，同时进行                           | 任务块在日历重叠 |

> **默认模式为串行**。并发需由任务认领人主动发起（见 §3.5）。

##### 并发数上限

- **每人同一时段内，并发任务数上限为 3**
- 并发计数口径：同一人在某工作日内，`scheduled_start ≤ 该日 ≤ scheduled_end` 的任务数量
- 超出上限时，系统**拒绝**执行并发设置，并给出明确提示

---

#### 3.2 队列模型（串行部分）

每位开发人员维护一个**个人任务队列**，队列按以下规则排序：

1. **优先级降序**：P0 > P1 > P2
2. **同优先级内**：按认领时间升序（先认领先处理）
3. **并发任务不占用队列位置**：并发任务与其关联的基准任务共享同一时间段，不排在串行队列后面

##### 特殊规则：不打断进行中任务

- P0 任务认领后，插入串行队列最前面
- **但不中断当前"进行中"的任务**：若当前有任务正在执行（状态为"进行中"），P0 新任务排在该任务之后（即，当前任务完成后立即处理 P0）
- 后续串行任务整体后移，重新计算排期

---

#### 3.3 配合人排期同步

##### 规则

当认领人为任务添加配合人后，配合人的排期与**认领人的排期保持一致**：

```
配合人.scheduled_start = 任务.scheduled_start（即认领人的 scheduled_start）
配合人.scheduled_end   = 任务.scheduled_end（即认领人的 scheduled_end）
```

##### 配合人的并发处理

- 配合任务与认领人任务的时间段完全相同，因此配合任务属于**并发任务**
- 系统在添加配合人时，自动检查该配合人在对应时间段内的并发任务数
- 若添加后**配合人并发数超过 3**，则拒绝添加，并提示：
  > `{配合人姓名} 在该任务时间段内并发任务数已达上限（3个），无法作为配合人。请调整任务排期或选择其他配合人。`
- 配合人自身串行队列中后续任务的排期**不受影响**（配合任务视为并发插入，不挤占串行队列时间）

##### 排期触发时机（含配合人）

| 事件                         | 对认领人的影响                       | 对配合人的影响                             |
|------------------------------|--------------------------------------|--------------------------------------------|
| 认领任务                     | 生成串行排期（scheduled_start/end）  | 暂无（配合人尚未添加）                     |
| 添加配合人                   | 无变化                               | 生成与认领人相同的排期，校验并发数         |
| 认领人任务排期变更           | 重算串行队列                         | 同步更新所有配合人的 scheduled_start/end   |
| 配合人被移除                 | 无变化                               | 移除该任务的排期记录，并发数-1             |
| 认领人任务提交/完成          | 从串行队列移除，后续任务前移         | 同步移除配合人的排期记录                   |

---

#### 3.4 排期计算规则

- 基于任务的"**拟投入人天**"计算时间跨度
- 自动排除周末（周六、周日）和法定节假日
- 每工作日按 1 人天计算
- **并发任务不叠加人天计算**：并发的两个任务各自占用对应的拟投入人天，不合并

---

#### 3.5 手动设置并发任务

任务认领人可以主动将**串行队列中靠后的任务**设置为并发，使其与当前或前面的任务同时进行。

##### 触发条件

- 操作人为任务认领人
- 目标任务处于已认领/进行中状态
- 目标任务当前为串行排期（非并发状态）

##### 交互流程

```
1. 认领人在「我的日程」或「我的任务列表」中，对目标任务点击「设为并发」
   ↓
2. 系统弹出【确认对话框】（见下方文案）
   ↓
3. 认领人点击「确认设为并发」
   ↓
4. 系统校验：认领人 + 所有配合人 在重叠时间段的并发数
   ↓
   ├── 校验通过 → 执行并发，更新排期，后续串行任务前移
   └── 校验失败 → 提示冲突人员，操作中止
```

##### 确认对话框文案

```
⚠️ 设置并发任务确认

您即将将「{任务名称}」设置为与「{基准任务名称}」并发执行。

并发意味着这两个任务将在同一时间段同时推进。
在设置并发前，请确认：

  ✅ 您已与本任务的所有配合人（{配合人列表，如有}）完成排期协调
  ✅ 您已了解并发后两个任务的人天投入将同时计算

并发后的预期排期：
  · {任务名称}：{new_scheduled_start} ～ {new_scheduled_end}
  · 原串行队列中后续任务整体前移

[取消]   [确认设为并发]
```

##### 并发数超限提示

当设置并发后，认领人或任一配合人的并发数将超过 3 时，**操作中止**，弹出提示：

```
⛔ 无法设置并发

以下人员在任务重叠时间段内的并发任务数将超出上限（3个）：

  · {姓名A}：当前 3 个并发任务，无法再增加
  · {姓名B}：当前 3 个并发任务，无法再增加

请与相关配合人确认排期，或变更配合人后再尝试设置并发。
```

##### 并发设置后的排期影响

- 目标任务的 `scheduled_start` / `scheduled_end` 更新为与基准任务重叠的时间段
- 目标任务从串行队列中移除
- 串行队列中后续任务整体前移，重新计算排期
- 目标任务的所有配合人排期同步更新

##### 取消并发

- 认领人可以对已设为并发的任务点击「取消并发」
- 取消后，该任务重新插入串行队列（按优先级+认领时间插入）
- 后续任务重新计算排期

---

#### 3.6 排期结果字段（更新）

每个任务认领/设置并发后，维护以下排期字段：

| 字段                  | 含义                                               |
|-----------------------|----------------------------------------------------|
| `scheduled_start`     | 预计开始工作日（YYYY-MM-DD）                       |
| `scheduled_end`       | 预计完成工作日（YYYY-MM-DD）                       |
| `queue_position`      | 在认领人串行队列中的位置（并发任务此字段为 NULL）  |
| `is_concurrent`       | 是否为并发任务（Boolean）                          |
| `concurrent_with`     | 并发基准任务 ID（仅并发任务有值）                  |

配合人维度新增字段（存储在 `task_collaborators` 关联表）：

| 字段                  | 含义                                               |
|-----------------------|----------------------------------------------------|
| `scheduled_start`     | 与任务认领人相同的开始日期                         |
| `scheduled_end`       | 与任务认领人相同的结束日期                         |

---

#### 验收标准

- [ ] 同优先级任务按认领时间排序

- [ ] 前置任务完成后，后续串行任务排期自动前移更新

- [ ] 认领人排期变更后，所有配合人排期同步更新

### FR-04 个人日程视图（工作台）

#### 描述

在**个人工作台**页面新增「**我的日程**」Tab 或卡片，以**月历/周历**形式直观展示该开发人员所有已认领任务的排期。

#### 功能点

##### 4.1 视图切换

- 支持**月视图**（默认）和**周视图**切换
- 支持前后翻页查看历史和未来排期

##### 4.2 任务块展示

- 在日历对应日期格上展示任务色块，信息包含：
  - 任务名称（截断显示，hover 展开）
  - 优先级色标（P0 红、P1 橙、P2 灰）
  - 若一天有多个任务块（理论上不应出现，但兜底），堆叠展示

##### 4.3 今日高亮

- 今日所在格加粗/高亮边框
- 当前"进行中"任务显示特殊标识（如 🔵 进行中）

##### 4.4 任务点击

- 点击任务块，弹出侧抽屉或 Popover，展示任务详情（标题、优先级、拟投入人天、截止时间、状态）
- 提供「查看详情」快捷入口，跳转至任务详情页

##### 4.5 时间范围说明

- 已认领且未完成的任务：显示 `scheduled_start` ～ `scheduled_end`
- 已完成任务（已提交/已确认）：以实际完成时间显示，灰度处理
- 过期未完成（超过 `scheduled_end` 但仍在进行中）：以红色边框标识延误

### FR-05 项目日程视图（进展数据页）

#### 描述

在**项目管理 → 进展数据**页面新增「**项目日程**」Tab，以**甘特图风格**的时间线视图展示该项目所有任务的排期分布。

#### 功能点

##### 5.1 时间线视图（Gantt 风格）

- 横轴：时间（天/周）
- 纵轴：任务列表（按优先级排序，P0 在上）
- 任务条显示：任务名称、负责人、优先级色标、状态
- 支持按月/季度缩放时间轴

##### 5.2 任务状态着色

| 状态           | 条形颜色     |
|----------------|--------------|
| 已认领/进行中  | 蓝色（实心） |
| 已提交/已确认  | 绿色（实心） |
| 已发布（无排期）| 虚线框      |
| 延误（超期）   | 红色（实心） |

##### 5.3 筛选与分组

- 按**负责人**分组折叠
- 按**优先级**筛选（P0/P1/P2）
- 按**状态**筛选
- 支持关键字搜索任务名称

##### 5.4 今日标线

- 在时间轴上绘制「今日」竖线，方便判断进度

##### 5.5 信息展示

- 悬浮任务条，显示：任务名、负责人、优先级、拟投入人天、scheduled_start、scheduled_end、当前状态

## 数据模型变更

### Task 表新增字段

| 字段名                | 类型                          | 默认值  | 说明                                               |
|-----------------------|-------------------------------|---------|---------------------------------------------------|
| `priority`            | ENUM('P0','P1','P2')          | 'P2'    | 任务优先级                                         |
| `priority_multiplier` | DECIMAL(4,2)                  | 1.00    | 溢价系数（冗余存储，便于查询）                     |
| `scheduled_start`     | DATE                          | NULL    | 预计开始日期（认领时生成）                         |
| `scheduled_end`       | DATE                          | NULL    | 预计完成日期（认领时生成）                         |
| `queue_position`      | INTEGER                       | NULL    | 在认领人串行队列中的位置；并发任务此字段为 NULL    |
| `is_concurrent`       | BOOLEAN                       | FALSE   | 是否为并发任务                                     |
| `concurrent_with`     | INTEGER (FK → tasks.id)       | NULL    | 并发基准任务 ID，仅并发任务有值                    |

### task_collaborators 表新增字段

> 配合人关联表（原有），新增排期字段：

| 字段名            | 类型    | 默认值 | 说明                                       |
|-------------------|---------|--------|--------------------------------------------|
| `scheduled_start` | DATE    | NULL   | 与任务认领人相同的预计开始日期             |
| `scheduled_end`   | DATE    | NULL   | 与任务认领人相同的预计完成日期             |

### 节假日配置表（新增）

| 字段名       | 类型    | 说明            |
|--------------|---------|-----------------|
| `id`         | INTEGER | 主键            |
| `date`       | DATE    | 节假日日期      |
| `name`       | VARCHAR | 节假日名称      |
| `year`       | INTEGER | 所属年份（索引）|

> 初期可内置 2025-2026 年法定节假日数据，后续提供管理员维护界面（P2 功能）。

### 产值计算逻辑变更

原有 `task_value` 计算需更新为：

```python
task_value = actual_days * developer_daily_rate * priority_multiplier
```

### 新增接口

| 接口                                          | 方法  | 说明                                           |
|-----------------------------------------------|-------|------------------------------------------------|
| `GET /users/me/schedule`                      | GET   | 获取当前用户排期（按日期范围，含并发任务）     |

```json
      "concurrent_with": 123,

```

#### `POST /tasks/{id}/set-concurrent` 请求体

```json
{
  "concurrent_with_task_id": 123
}
```

响应：
- `200`：并发设置成功，返回更新后的排期信息
- `422`：并发数超限，返回超限人员列表

```json

  "exceeded_users": [
    { "user_id": 1, "name": "张三", "current_concurrent": 3 },

```

## UI/UX 设计要点

1. **溢价提示要醒目但不打扰**：使用内联 Warning 提示，而非弹窗打断操作
2. **日历组件复用**：个人日程和项目日程共用同一套基础日历/甘特组件，减少重复开发
3. **甘特图初期不要求拖拽编辑**：只读展示，避免引入排期手动修改的复杂度
4. **颜色体系统一**：P0=红(#F56C6C)、P1=橙(#E6A23C)、P2=灰(#909399)，复用 Element Plus 语义色
5. **移动端兼容**：日历视图在小屏幕下退化为列表视图（周视图），不强制横向滚动

## 风险与约束

| 风险                                   | 影响 | 缓解措施                                                             |
|----------------------------------------|------|----------------------------------------------------------------------|
| 排期算法复杂度高                       | 高   | 先单测覆盖，边界条件（跨年、全节假日周、并发+串行混合）必须测试     |
| 甘特图组件选型                         | 中   | 优先用 ECharts 自定义甘特，避免引入重量级第三方库                    |
| 节假日数据维护成本                     | 低   | 内置当年数据，管理员界面留到 P2 阶段实现                             |
| P0 插队逻辑与"置顶"功能重叠           | 中   | P0 优先级排队 ≠ 手动置顶；置顶可打断进行中任务，P0 不打断           |
| 产值计算变更影响现有数据               | 中   | 历史已确认任务的 `priority_multiplier` 默认设为 1.00，不影响历史数据 |
| 并发数统计的并发安全问题               | 高   | 设置并发操作需加数据库行锁或事务隔离，避免两人同时设置导致超限绕过  |
| 配合人排期联动更新范围大               | 中   | 认领人排期变更时，批量更新配合人排期需异步处理，避免阻塞接口响应     |
| 并发任务在日历中视觉叠加难处理         | 中   | 前端月视图单日最多显示 3 条，超出折叠为"+N 个任务"，点击展开         |
| "设为并发"的基准任务选择交互不清晰    | 中   | 初期限制：只允许将目标任务设为"与当前进行中任务并发"，降低选择复杂度 |
